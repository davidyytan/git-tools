# Contributing to Git Tools

## Before You Start

Read these first:

- [README](README.md)
- [Git Flow Guide](docs/git-flow-guide.md)

`git-tools` supports multiple LLM providers, prompt files, workflow examples, and a Git Flow-oriented release model. Keep changes aligned across code, docs, prompts, and tests.

## Development Setup

```bash
git clone https://github.com/<your-user>/git-tools.git
cd git-tools

uv tool install -e .
```

Configure a provider in one of these ways:

```bash
git-tools config
```

or:

```bash
cp git-tools.env.example git-tools.env
```

Provider and model defaults come from `mappings.json` when present, otherwise `mappings.json.example`.

If you want local repo-specific defaults:

```bash
cp mappings.json.example mappings.json
```

`git-tools config` writes user overrides to `~/.config/git-tools/config.env`. It does not edit `mappings.json`.

## Branch and PR Flow

Normal code changes should start from `develop`:

```bash
git checkout develop
git pull origin develop
git checkout -b feature/<name>
```

or:

```bash
git checkout develop
git pull origin develop
git checkout -b bugfix/<name>
```

Use the special branches only for the Git Flow release process:

- `release/*`
- `hotfix/*`
- `sync/*`

The recommended day-to-day policy is:

- normal PRs into `develop` use Conventional Commit titles
- `release/* -> master` uses `git-tools pr --release-pr`
- `hotfix/* -> master` uses `git-tools pr --hotfix-pr`
- `sync/* -> develop` uses `git-tools pr --sync-pr`

For the full release policy, branch naming, and merge settings, use [Git Flow Guide](docs/git-flow-guide.md).

## Commits

Use Conventional Commit headers for normal code changes:

```text
type(scope): description
```

Common types:

- `feat`
- `fix`
- `docs`
- `style`
- `refactor`
- `test`
- `chore`
- `ci`

For normal feature and bugfix work, prefer using the tool:

```bash
git add -A
git-tools commit
```

Do not use Conventional Commit-style release titles for release, hotfix, or sync PRs. Those PRs use fixed titles generated by `git-tools pr`.

## Pull Requests

Typical commands:

```bash
git-tools pr --base develop
git-tools pr --base master --release-pr
git-tools pr --base master --hotfix-pr
git-tools pr --base develop --sync-pr
```

Expected PR titles:

- normal develop PRs: Conventional Commit title
- release PRs: `Release x.y.z`
- hotfix PRs: `Hotfix x.y.z`
- sync PRs: `Sync x.y.z`

PR output should not include `## Related Issue` or `Issue: #...` placeholders.

## Validation

Before opening a PR, run:

```bash
uv run python -m unittest discover -s tests
git diff --check
```

If you change provider configuration, prompt files, or workflow examples, also check the related docs and tests.

## Keeping Things In Sync

When you change one of these areas, update the others in the same PR when needed:

- provider config in `git_tools/config/`
- provider defaults in `mappings.json.example`
- user-facing env examples in `git-tools.env.example`
- prompt files in `git_tools/prompts/`
- workflow examples in `.github/workflows/`
- docs in `README.md` and `docs/git-flow-guide.md`

If the change affects CLI behavior, add or update tests in `tests/`.
